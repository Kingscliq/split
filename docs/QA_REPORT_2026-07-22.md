## Bugs and Release Risks

### Critical

No critical defect was directly reproduced.

### High

#### QA-H1 — Real signed create-and-pay workflow is unverified

**Reproduction:** Connect two funded Freighter Testnet wallets, create an XLM Split as the creator, then pay it as the participant.

**Expected:** Both transactions succeed, XLM moves directly to the creator, and the UI refreshes to paid/completed.

**Actual:** Not executed because no wallet authorization was granted during this QA run.

**Impact:** The application cannot be declared end-to-end complete or ready for a public Testnet demo.

#### QA-H2 — Production dependency audit reports unresolved high-severity advisories

Affected installed packages include:

- `axios@1.16.1`, through `@stellar/stellar-sdk@16.0.1`
- `sharp@0.34.5`, through `next@16.2.10`
- `postcss@8.4.31`, through `next@16.2.10` (moderate)

`npm audit --omit=dev` reported 5 total vulnerabilities: 3 high and 2 moderate. The suggested `--force` resolution proposes breaking dependency changes and must not be applied without compatibility review.

**Expected:** No unresolved high-severity production advisories, or a documented risk acceptance with proven non-exploitability.

**Actual:** Audit exits with code 1.

### Medium

#### QA-M1 — QR-code sharing is missing

**Reproduction:** Open any Split detail page and inspect sharing actions.

**Expected:** Copy link, WhatsApp, and a visible/scannable QR code.

**Actual:** Only Copy link and WhatsApp are implemented.

#### QA-M2 — Activity navigation is hardcoded to Split `12`

**Reproduction:** Select Activity in the sidebar.

**Expected:** A real activity/history destination or a disabled/unavailable state.

**Actual:** Navigation always targets `/split/12`; the deployed contract currently has zero Splits.

#### QA-M3 — Event behavior is not regression-tested

**Reproduction:** Review `contracts/split_contract/src/test.rs`.

**Expected:** Assertions for `SplitCreated`, `SharePaid`, `SplitCompleted`, and `SplitClosed` topics and payloads.

**Actual:** Events are published by contract code, but no test reads or asserts them.

#### QA-M4 — Contract storage TTL is not maintained

**Reproduction:** Review all persistent storage reads/writes in the contract.

**Expected:** A documented TTL extension/restoration policy for long-lived Splits.

**Actual:** No `extend_ttl` or restoration handling exists. Old Split records can eventually require archival restoration.

#### QA-M5 — Wallet account/network changes are not observed automatically

**Reproduction:** Connect Freighter, then switch account or network without reloading or pressing the wallet button again.

**Expected:** The connected address/network state updates immediately.

**Actual:** `WalletContext` checks on initial mount and explicit connect only; no wallet-change watcher is registered.

#### QA-M6 — Transaction submission statuses are not fully distinguished

**Reproduction:** Make RPC return `TRY_AGAIN_LATER` or a non-submitted response from `sendTransaction`.

**Expected:** Retryable, duplicate, rejected, and pending states receive accurate handling.

**Actual:** Only `ERROR` is handled immediately. Other statuses enter transaction polling and can end with a misleading pending message.

#### QA-M7 — No frontend automated test suite

**Reproduction:** Inspect `frontend/package.json`.

**Expected:** Automated tests for amount conversion, form validation, wallet errors, contract decoding, and transaction states.

**Actual:** Only dev, build, start, and lint scripts exist.

#### QA-M8 — Large token values are converted through JavaScript `Number`

**Reproduction:** Format or preview sufficiently large valid `i128` amounts.

**Expected:** Exact decimal formatting from `bigint` base units.

**Actual:** `formatAmount`, per-participant preview, and waived preview convert `bigint` to `Number`, which can lose precision for large values.

#### QA-M9 — Invalid and missing Split pages return HTTP 200

**Reproduction:** Request `/split/abc` or `/split/999` directly.

**Expected:** A semantic HTTP 404 for invalid/nonexistent public resources.

**Actual:** Server responds 200, then client JavaScript renders an error message.

### Low

#### QA-L1 — Copy-link action has no success or failure feedback

**Expected:** A “Link copied” confirmation or recoverable clipboard error.

**Actual:** Clipboard promise is awaited without visible feedback or error handling.

#### QA-L2 — Dashboard “Splits created” count is capped by the recent-list limit

**Reproduction:** Create more than 12 Splits.

**Expected:** The headline total reflects `get_split_count` while the list remains limited to 12.

**Actual:** The headline displays `splits.length`, so it cannot exceed 12.

#### QA-L3 — Nonparticipant payment control remains clickable

**Reproduction:** Open a real active Split with a connected wallet that is not a participant.

**Expected:** “Not a participant” control is disabled.

**Actual:** The label changes, but the disabled expression does not disable the button for a connected nonparticipant; clicking produces an error.

The following require user-controlled wallet actions and were not executed:

- Freighter permission grant
- Wrong-network prompt behavior
- Locked wallet behavior
- Signature rejection
- Insufficient XLM and USDC balances
- Real XLM creation and payment
- Real USDC creation and payment
- Creator balance verification
- Contract balance verification after payment
- Multi-wallet completion
- Creator close and unauthorized close attempt
- Transaction hashes and explorer evidence

## Release Recommendation

Do not declare the integration complete or publish the public Testnet demo yet.

Minimum release gates:

1. Complete one real signed XLM create-and-pay flow with two wallets and record transaction hashes.
2. Verify the creator receives the exact amount and the Split contract retains no funds.
3. Repeat the payment path with configured testnet USDC, including trustline/balance handling.
4. Add QR-code sharing.
5. Resolve or formally assess the high-severity production dependency advisories.
6. Add contract event assertions and the missing boundary tests.
7. Commit the frontend implementation and `.env.example`; keep `.env.local` and `.env.deploy` untracked.

After those gates, rerun the signed transaction, responsive, accessibility, and dependency checks for a final Pass/Fail decision.
